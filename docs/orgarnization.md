# setlist-maker 에이전트 조직도 (v2)

> v1을 Claude Code의 실제 실행 모델에 맞게 재설계. PRD(`docs/PRD.md`) 파일럿 범위(§4) 기준.

## 0. 실행 모델 매핑 (이 조직도가 실제로 돌아가는 방식)

| 조직도 개념 | Claude Code 구현 |
|---|---|
| `부장` | 메인 세션 그 자체 (사용자와 직접 대화하는 Claude) |
| 직원 고용 | `Agent` 툴로 서브에이전트 스폰 (`model` 파라미터로 직급별 모델 지정) |
| 보고 / 재작업 지시 / 검수 왕복 | `SendMessage`로 기존 에이전트의 컨텍스트를 유지한 채 대화 계속 |
| 결재 요청 | 메인 세션이 사용자에게 질문·확인 (AskUserQuestion 등) |
| 코드 검수 | `/code-review` 스킬 + 팀장 에이전트 검수 |
| git 권한 | 부장(메인 세션) 전담 — 서브에이전트는 커밋·푸시 금지 |

**중요한 제약 — 콜드 스타트**: 서브에이전트는 스폰 시점에 이전 대화 맥락을 전혀 모른다.
따라서 부장은 모든 업무 할당 시 아래 5가지를 반드시 프롬프트에 포함해야 한다:

1. 티켓 목표 (한 문장)
2. 근거가 되는 PRD 섹션 번호
3. 허용 작업 경로 (3번 표 참조)
4. 완료 판정 기준 (acceptance criteria)
5. 보고 형식 (규칙 R4)

## 1. 규칙

- **R1 (지휘 체계)**: 상위 레벨은 하위 레벨에게 업무를 할당·명령할 수 있다.
  하위 레벨은 상위 레벨에게 건의할 수 있으며, 건의는 정량적 데이터에 입각해야 한다.

- **R2 (검수)**: 사원의 작업 결과물은 해당 팀장이 검수하여 승인/반려(코멘트와 함께 재작업
  요구 가능)를 결정한다. 검수 왕복은 SendMessage로 수행한다(에이전트 재스폰 금지 — 컨텍스트
  낭비). 팀장의 결과물은 부장이 검수하며, 코드 변경분은 부장이 `/code-review`를 병행한다.

- **R3 (정직한 보고)**: 모든 직책은 완료한 것은 완료했다고, 완료하지 못한 것은 사유와 함께
  완료하지 못했다고 보고한다. 테스트 실패·미검증 상태를 "완료"로 포장하는 것을 금지한다.

- **R4 (보고서)**: 사원·팀장의 보고서는 에이전트의 최종 메시지가 원본이며, 부장이 이를
  `document-archive` 브랜치의 `archive/reports/YYYY-MM-DD-<티켓명>.md`로 정리해 남긴다(main과
  무관한 별도 커밋). 형식: 목표 / 수행 내용 / 완료·미완료 목록 / 미완료 사유 / 다음 단계 제안.
  직책마다 별도 보고서 파일을 중복 생성하지 않는다.

- **R5 (토큰 게이트)**: 부장은 **에이전트를 스폰하기 전에** 토큰 게이트 툴(4번 참조)을 1회
  실행하여 `(세션 ≥ 80%) or (주간 ≥ 90%) or (Fable 주간 ≥ 70%)`이면 신규 스폰을 중단하고,
  남은 작업을 R4 형식의 보고서로 기록한 뒤 사용자에게 보고하고 지시를 기다린다.
  - v1은 "모든 직책이 각자 체크"였으나, 서브에이전트가 각자 체크하면 그 자체가 토큰 낭비이고
    측정 대상도 동일하므로 **부장 단일 체크포인트**로 변경한다.
  - 툴이 제작되기 전까지 이 게이트는 사용자가 `/usage`로 수동 확인하는 것으로 갈음한다.

- **R6 (경로 권한)**: 각 팀은 3번 표에 지정된 경로 내에서만 파일을 읽기·쓰기·편집한다.
  지정 경로 밖의 수정이 필요하면 작업을 멈추고 보고서에 사유를 적어 부장에게 요청한다.
  - 이 권한은 파일시스템 수준에서 강제되지 않으므로(프롬프트 규칙일 뿐), 부장은 검수 시
    `git status`/diff로 **경로 위반 여부를 반드시 확인**한다. 위반 발견 시 반려.
  - git 커밋·푸시는 부장 전담. 서브에이전트는 git 쓰기 명령을 실행하지 않는다.

- **R7 (경합 방지)**: 같은 파일을 건드리는 티켓은 동시에 스폰하지 않는다. 부장은 작업 경로가
  겹치지 않는 티켓만 병렬로 스폰하고, 겹치는 티켓은 순차 처리한다. 경합이 사후 발견되면
  부장이 조율하여 재작업을 지시한다.

- **R8 (토큰 절약)**: 정량적으로 검증되지 않은 사실로 의문을 제기하거나 순환 논리 등 불필요한
  토큰 사용을 지양한다. 단순 조사·검색 티켓은 사원 스폰 대신 `Explore` 에이전트(읽기 전용,
  경량)를 사용한다.

- **R9 (고용 한도)**: 팀당 동시 활성 사원은 최대 3명. 전체 동시 활성 서브에이전트는 최대
  4개(모니터링 가능 한도)로 제한한다.

- **R10 (멀티 로컬 운영)**: 작업은 두 대의 로컬 — 기기 A(샘플 데이터만 보유, 현재 기기)와
  기기 B(전체 오디오 캐시 보유) — 를 오가며 진행한다. 동기화 채널은 GitHub 원격
  (`origin/main`)이 유일하다.
  - **기기 전환 전**: 부장은 검수 완료분을 커밋·푸시하고, 진행 중·미완료 작업을 R4 보고서로
    남긴다. **전환 후 첫 작업**: `git pull` + `document-archive` 브랜치의 `archive/reports/` 최신
    보고서 확인으로 상태 복원(예: `git show document-archive:archive/reports/...`).
  - 오디오 원본이 필요한 작업(피처 재추출, key 정확도 검증 등)은 **기기 B 전용 티켓**으로
    표시하고, 산출물은 피처 CSV/JSON만 커밋한다(오디오 파일은 .gitignore로 차단).
  - 에이전트 세션과 토큰 게이트 집계는 기기 로컬이며 기기 간 이전되지 않는다 — 토큰 게이트는
    해당 기기의 사용량만 근사하므로, 두 기기를 오가는 날은 게이트 수치를 보수적으로 해석한다.

- **R11 (소스 경계·폴더 규칙)**: 모든 소스코드 작업은 **`./src/` 안에서만** 수행한다
  (`data/`는 데이터 산출물, `docs/`는 문서 전용 — 코드 금지). 작업 성격·폴더별 작성규칙은
  각 폴더의 `README.md`에 둔다: 모든 직원은 폴더에서 작업을 시작하기 전 해당 폴더의
  `README.md`를 먼저 읽고 따르며, 새 폴더에 고유한 작성규칙이 필요해지면 그 폴더의
  `README.md`를 만들어 규칙을 남긴다. README와 티켓 명세가 충돌하면 멈추고 부장에게
  보고한다(R6 준용). 전체 폴더 맵은 `src/README.md` 참조.

## 1.5 사원 모델 선정 기준 (부장이 스폰 시 결정)

사원의 기본 모델은 **haiku**로 하고, 아래 조건 중 하나라도 해당하면 **sonnet**으로 상향한다.

- **haiku 충분 (기본)**: 팀장 명세가 완결적(입출력 포맷·검증 기준·엣지 케이스가 티켓에 전부
  명시)이고, 산출물이 단일 모듈 수준이며, 실패가 단위 테스트로 즉시 검출되는 작업.
  예: 매핑 테이블 구현, URL 파서, 정해진 스키마의 파일 변환·복사.
- **sonnet 상향 조건**: ① 구조가 미지인 데이터·시스템 탐사가 필요(예: 토큰 게이트의 JSONL
  구조 역공학) ② 다중 파일 통합·아키텍처 판단 여지 존재 ③ 디버깅 루프가 예상되는 작업
  ④ haiku 산출물이 반려 2회를 넘긴 경우(모델 한계로 판단).
- 팀장(opus)·부장 모델은 변경 없음 — 검수·설계 품질이 전체 병목이므로 유지한다.

## 3. 팀별 작업 경로

| 직책 | 허용 경로 (읽기+쓰기) | 비고 |
|---|---|---|
| 부장 | 리포지토리 전체 + git | 유일한 git 쓰기 권한 |
| 데이터팀 | `data/`, `src/scripts/` | 원본 CSV/JSON 사본과 가공 스크립트 |
| 코드설계팀 | `src/backend/`, `src/frontend/`, `src/tests/` | 도메인·어댑터·API·UI |
| R&D팀 | 로컬 작업 후 `document-archive` 브랜치 `archive/research/` (쓰기, 부장 경유), `data/` (읽기 전용) | 연구보고서 산출 |
| 공통 읽기 | `docs/PRD.md`, `CLAUDE.md`, `docs/orgarnization.md` | 모든 직책 읽기 가능 |

- `src/scripts/`는 존재하며(구 `scripts/`에서 2026-07-10 이전), `src/backend/`·`src/frontend/`·
  `src/tests/`는 README만 있는 상태 — 첫 구현 티켓에서 이 구조대로 생성한다(R11).
- 외부 레포(`bandori-song-sorter`)는 읽기 전용이며, 필요한 파일은 데이터팀이 `data/`로 복사해
  이 레포를 단일 소스로 만든다(원본 레포 수정 금지).

## 4. 토큰 게이트 툴 (신규 제작 — 데이터팀 첫 티켓 후보)

> **[2026-07-10 폐기 예고]** 아래 트랜스크립트 집계(token seeking) 방식은 세션 축 오판이
> 실측으로 확인되어(스크립트 2.0% vs `/usage` 85%) 사용자 결정으로 폐기 예정. **MCP 기반
> 관찰(observation) 도구로 전환**한다 — 설계·사유는 `document-archive` 브랜치의
> `archive/reports/2026-07-10-token-gate-mcp-transition.md` 참조. 전환 완료 시 이 절과 R5를
> 개정할 것. 전환 전까지 게이트 수치는 참고용으로만 쓰고 `/usage` 실측을 우선한다.

R5 게이트가 자동으로 동작하려면 사용량 조회 툴이 필요하다. Claude Code는 대화 트랜스크립트를
`~/.claude/projects/**/*.jsonl`에 남기며 각 메시지에 usage(토큰 수) 필드가 포함되므로, 이를
집계해 근사치를 산출할 수 있다(커뮤니티 툴 ccusage와 동일한 방식).

- **위치**: `src/scripts/token_gate.py` (또는 독립 CLI)
- **출력 계약**(부장이 파싱하는 인터페이스):

```json
{
  "session_pct": 42.0,
  "weekly_pct": 61.5,
  "fable_weekly_pct": 55.0,
```
