# Version 브랜치 — README.md 작성 규칙

`README.md`(버전 로그)에 새 항목을 추가할 때 따르는 형식·분량 규칙이다. **언제 태깅하고
언제 이 브랜치에 기록하는지**(브랜치 종류별 트리거)는 `main`의 `git-rules.md` "버전
태깅(Tagging) 및 배포 규칙" 절이 원칙이다 — 이 문서는 그 원칙 위에서 **항목을 실제로 어떻게
쓰는지**만 다룬다.

## 정렬

**항목은 최신 버전이 맨 위, 오래된 버전이 아래로 가도록 정렬한다.** 새 항목은 직전 최신
항목(`---` 구분선 다음) 바로 위에 삽입한다 — 파일 끝에 이어붙이지 않는다.

`Beta`(v1.0.0 이전 커밋 단위 이력)와 `v1.0.0`은 가장 오래된 항목이라 파일 맨 끝에 고정한다.
`Beta` 절 내부는 예외적으로 날짜 오름차순(오래된 순)을 유지한다 — 커밋 단위라 개별 버전
헤더가 없어 전체 정렬 규칙과 별개로 다룬다.

## 항목 형식

```
## vX.Y.Z — YYYY-MM-DD

<요약, 100자 이내>

- 기준 커밋: `abcdef1`
- 관련 PR: [#N](https://github.com/sbb2002/bandori-playlist-maker/pull/N)
```

- **요약은 100자 이내**로 쓴다. "무엇을 왜 바꿨는지"를 한 줄로 압축한다 — 구현 세부(함수명,
  코드 경로, 조사 과정, 재태깅 사유 등)는 담지 않는다. 그런 맥락은 실제 PR·커밋 diff에
  남아있으니 거기서 찾으면 된다. 이 로그는 "무슨 일이 있었는지"를 빠르게 훑기 위한 색인이다.
- **명사형 종결("~함", "~수정", "~추가") 등으로 문장을 짧게 줄여도 된다.** 완전한 문어체
  문장일 필요 없다.
- `기준 커밋`·`관련 PR`은 그대로 유지한다(분량 제한 대상 아님 — 메타데이터).
- PR 오픈 전이라 커밋·PR 번호가 아직 없으면 `(PR 오픈 직전 커밋)` / `(PR 오픈 후 갱신)`처럼
  플레이스홀더를 쓰고, 나중에 실제 값으로 교체한다.

## 하지 않는 것

- 조사·시행착오·기각한 대안 등 "과정" 서술 — 요약 100자에는 결과만 담는다.
- 재태깅 이력("원래 vX.Y.Z였으나...") 같은 메타 코멘트 — git 태그 히스토리로 이미 추적 가능.
- 여러 PR·이슈를 한 항목에 몰아 나열 — 항목 하나는 버전 하나(=PR 하나)에 대응한다.

## 알려진 공백

2026-07-20 기준, `v1.9.0` 이후 `v1.10.0`·`v1.10.1`·`v1.11.0` 항목이 로그에 없다(태그는
존재하나 이 파일에 기록이 안 됨 — 어느 세션에서 기록을 빼먹은 것으로 추정). 세 버전의 실제
변경 내용을 조사해 채워 넣는 작업이 남아있다.
