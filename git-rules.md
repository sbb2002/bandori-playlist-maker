# Git Branch Workflow

```
> Merge 정책(누가 언제 `main`에 Merge하는지)은 `CLAUDE.md`의 "Working agreement"가 원칙이다.
> 이 문서는 그 원칙 위에서 **브랜치 종류별 생성·재사용 규칙, 영역 격리에 따른 Rebase 정책, 그리고 배포 태깅 규칙**을 다룬다.
```

## Branch Types

### `epic/*`

* 대규모 기능 개발 및 구조 개편용 브랜치 (유즈케이스 Core 레벨 수정, UI/UX 전면 개편 등)
* 항상 최신 `main`에서 생성한다.
* 거대한 마일스톤이나 파괴적 변경(Breaking Changes) 단위당 브랜치 하나를 사용한다.
* `main`에 변경 사항이 추가되었다면 Merge 전에 최신 `main`으로 Rebase한다.
* **[배포 및 태깅]** PR을 열기 직전(작업 마지막 커밋 시점)에 **Major(X) 버전을 올리는 태깅**(`v1.2.3` ➡️ `v2.0.0`)과 `versionlog.md` 기록을 브랜치 안에서 끝낸다 — 머지 시점이 아니라 PR을 보낼 때 이미 버전이 올라간 상태여야 한다("버전 태깅(Tagging) 및 배포 규칙" 절 참고).

---

### `feature/*`

* 새로운 기능 개발 및 고도화용 브랜치 (어댑터 및 인프라 수정, UI 일부 수정, 피쳐 스펙 및 파라미터/Threshold 튜닝 등)
* 항상 최신 `main`에서 생성한다.
* 기능 하나당 브랜치 하나를 사용한다.
* `main`에 변경 사항(`hotfix/*` 등 다른 브랜치의 머지)이 추가되었다면 Merge 전에 최신 `main`으로 Rebase한다.
* **[배포 및 태깅]** PR을 열기 직전(작업 마지막 커밋 시점)에 **Minor(Y) 버전을 올리는 태깅**(`v1.0.0` ➡️ `v1.1.0`)과 `versionlog.md` 기록을 브랜치 안에서 끝낸다 — 머지 시점이 아니라 PR을 보낼 때 이미 버전이 올라간 상태여야 한다("버전 태깅(Tagging) 및 배포 규칙" 절 참고).

---

### `hotfix/*`

* 버그 수정용 브랜치 (스펙 변경 없이 주로 `src/` 디렉터리 내부 단순 디버깅)
* 항상 최신 `main`에서 생성한다.
* 버그 하나당 브랜치 하나를 사용한다.
* 수정 완료 후 가능한 한 빨리 PR을 연다(긴급도가 높으므로 리뷰 우선순위도 가장 높다).
* `main`에 다른 브랜치가 머지되면 최신 `main`으로 즉시 Rebase하여 충돌을 방지한다.
* **[배포 및 태깅]** PR을 열기 직전(작업 마지막 커밋 시점)에 **Patch(Z) 버전을 올리는 태깅**(`v1.0.0` ➡️ `v1.0.1`)과 `versionlog.md` 기록을 브랜치 안에서 끝낸다 — 머지 시점이 아니라 PR을 보낼 때 이미 버전이 올라간 상태여야 한다("버전 태깅(Tagging) 및 배포 규칙" 절 참고).

---

### `data`

* 데이터 주입 자동화 전용 단일 브랜치 (`data/` 디렉터리만 수정)
* 임의의 주기로 데이터가 자동 커밋 및 푸시된다. 커밋에 PR을 요구하지 않는다(자동화 봇 예외).
* **[중요]** 자동화 스크립트(봇)의 싱크 유지를 위해 **절대로 Rebase를 수행하지 않는다.**
* **[2026-07-15 변경] `main`에 병합되지 않는 완전히 독립된 브랜치다.** `main`은 앱 소스만
  배포하고(`data/` 디렉터리를 담지 않음), 배포된 backend가 런타임에 이 브랜치의
  `data/songs_master.csv`를 직접 원격 fetch해 쓴다(`src/backend/app/repo/remote_source.py`,
  기동 시 + 주기 리프레시). 데이터 갱신마다 `main`을 재배포(=서비스 프리징)하지 않기 위한
  설계 — `main` 머지는 여전히 PR 필수라는 일반 원칙(`CLAUDE.md` Working agreement)은 그대로이며,
  `data`는 애초에 `main`을 대상으로 하지 않으므로 그 원칙과 충돌하지 않는다.
* **[배포 및 태깅]** `main` 머지 자체가 없으므로 해당 없음(태깅 대상 아님).

---

### `document-archive` (구 `docs`)

* 아카이브 문서 전용 단일 브랜치 (`archive/` 디렉터리만 수정 — user-opinion, verification, reports,
  research, user_manual_pictures 등 "특정 시점 기록·다음 패치 아이디어" 티어. 티어 정의는 아래
  "코드/데이터 vs 문서 — 브랜치 분리 원칙" 참조).
* **`main`에 다시 PR/머지되지 않는, 완전히 분리된 별개 역사다.** `origin`에 직접 commit·push만
  한다 — PR을 열지 않는다.
* 머지는 안 하지만 **읽기 참조는 언제든 가능**하다: `git show document-archive:archive/<path>`로
  특정 파일 조회, 또는 `git worktree add`로 별도 폴더에 펼쳐 참고. 어느 쪽도 `main` 히스토리에
  영향 없음.
* 자잘한 브랜치 생성을 방지하고 아카이브를 깔끔하게 유지하기 위해 모든 아카이브 문서 작업은 이
  하나의 브랜치에서 진행한다. 작업 내용 및 추가된 문서는 커밋 히스토리를 통해 추적한다.
* `archive/ref/` 이하 문서(특히 `archive/ref/user-opinion/`)는 읽기 전용 참조 자료다 — 사용자의
  명시적 허락 없이 편집·삭제·이동·이름변경하지 않는다 (`CLAUDE.md` 문서 취급 규칙 참조).
* `main`과 완전히 분리되어 있으므로 Rebase가 불필요하다.
* **[주의]** `archive/research/` 경로는 `research/*` 브랜치가 산출한 연구 보고서의 최종 목적지다.
  `research/*`는 여전히 `main`에서 분기해 코드 실험을 진행하지만, 완료된 보고서 `.md`는 이 브랜치로
  별도 커밋해 반영한다(아래 `research/*` 절 참조).
* **[배포 및 태깅]** `main` 머지 자체가 없으므로 태깅 대상이 아니다(해당 없음).

---

### `research/*`

* 새로운 알고리즘, 모델, 분석 방법 등을 검증하기 위한 연구용 브랜치 (코드 실험 위주 작업)
* 항상 최신 `main`에서 생성한다.
* 하나의 연구 주제당 브랜치 하나를 사용한다.
* **Rebase 규칙:** `data`는 `main`에 병합되지 않으므로(위 `data` 절 참조) `main` 리베이스로는 최신
  데이터를 못 받는다. 최신 데이터가 필요하면 `git show data:data/songs_master.csv` 등으로 `data`
  브랜치를 직접 읽어온다(연구 브랜치 자체를 리베이스할 필요 없음).
* 기존 연구 툴을 활용하므로 `src/` 코드의 변경사항(`epic/*`, `feature/*`, `hotfix/*`)에 대해서는 강제적으로 Rebase할 필요가 없으며, 연구 프로세스에 맞춰 선택적으로 진행한다.
* 연구 브랜치 상호 간의 Rebase는 불필요하다.
* 실험 코드는 앱에 직접 반영되지 않아도 되며, 연구 결과가 유의미하지 않다면 저장소 소유자와 확인 후 브랜치를 삭제한다.
* 연구 결과가 채택되면 새로운 `feature/*` 브랜치를 생성하여 정식 기능(`src/`)으로 구현하며, `research/*` 브랜치를 그대로 `main` 대상 PR로 열지 않는다.
* **산출물 3종은 목적지가 각각 다르다:**
  1. **분석용 스크래치 데이터**(중간 계산·샘플 서브셋 등 연구 자체에만 필요한 것) — 브랜치 로컬에
     자유롭게 두고 정리하지 않는다. `research/*`는 어차피 `main`에 직접 머지되지 않으므로 여기 쌓인
     내용이 `main`을 오염시키지 않는다. 대용량 바이너리는 기존 `.gitignore` 규칙(`*.npy`/`*.npz`/
     `audio_cache/`)이 이미 걸러낸다.
  2. **검증돼서 정식 채택할 데이터**(예: `data/songs_master.csv`에 새 컬럼 추가) — `research/*`가
     직접 `data/`를 고쳐 `main`으로 보내지 않는다. 코드와 동일한 원칙으로, 채택된 데이터 변경은
     **새 `feature/*` 브랜치(또는 자동화라면 기존 `data` 브랜치)** 에서 다시 정식으로 반영한다.
     `research/*`는 "이 데이터가 이렇게 바뀌어야 한다"는 근거만 만드는 역할이다.
  3. **연구 보고서(.md)** — 연구 중에는 브랜치 로컬에 자유롭게 작성하되, 완료 시 그 `.md`만
     `document-archive` 브랜치의 `archive/research/`로 별도 커밋한다(코드가 `feature/*`로
     정식화되는 것과 별개 절차 — 문서는 `main`으로 가지 않는다).
* **[배포 및 태깅]** `main`에 직접 머지되지 않고 `feature/*` 브랜치를 새로 생성하여 정식 기능화 과정을 거치므로, 연구 브랜치 단계에서는 **태깅이 발생하지 않는다.**

## 코드/데이터 vs 문서 — 브랜치 분리 원칙

* **코드·데이터 변경**(로직 수정, CSV/데이터셋 수정 등)은 **항상 별도 브랜치를 새로 파고 PR도 새로 연다** — 리뷰어가 실제 동작을 검증해야 하므로 아카이브 브랜치와 섞이면 안 된다.
* **아카이브 문서 전용 변경**은 단일 `document-archive` 브랜치를 사용하여 커밋을 누적해 나간다(PR 없음, `origin`에 직접 push).
* 각 작업 영역은 디렉터리 단위로 격리되어 병행 작업이 가능하나, `research/*` 브랜치가 산출하는 연구
  보고서는 완료 시 `document-archive`의 `archive/research/`로 반영되므로 해당 산출물에 한해
  상호 컨텍스트를 동기화한다(위 `research/*` 절 참조).

### 문서 3분류(A/B/C)

| 티어 | 정의 | 목적지 |
|---|---|---|
| **A** | 설계 철학·최상위 규칙 문서 (앱 소스 이해에 필수) | `main` (루트 + `docs/`) |
| **B** | 직접 안 쓰이지만 개발자/방문자가 열람하는 문서 | `main` (`docs/`) |
| **C** | 특정 시점 기록·다음 패치 아이디어 (user-opinion, verification, reports, research, user_manual_pictures) | `document-archive`의 `archive/` 전용 |

## Merge Priority

브랜치의 우선순위는 다음과 같다 (PR 리뷰 요청 시 저장소 소유자에게 안내하는 순서).

1. `hotfix/*`
2. `feature/*`
3. `epic/*` (또는 `major/*`)

긴급 수정 PR은 가능한 한 먼저 검토·Merge되도록 우선 안내한다. `document-archive`는 `main`에
PR을 열지 않으므로 이 목록에 포함되지 않는다.

## Rebase & Version Tagging Policy (요약)

### 1. Rebase 정책
* 동일하게 `src/` 영역을 작업 중인 브랜치(`epic/*`, `feature/*`, `hotfix/*`)가 `main`에 Merge된 이후, 다른 작업 브랜치가 남아있다면 가능한 한 빠르게 최신 `main`으로 Rebase하여 충돌을 미리 해결한다.
* 기계(자동화 봇)가 제어하는 `data` 브랜치를 제외하고, 개발자가 만지는 브랜치 중 최신 코드/데이터 동기화가 필요한 시점에는 항상 `git fetch origin` 후 `git rebase origin/main`을 표준으로 삼는다. 폴더가 격리되어 있어 충돌 없이 안전하게 최신 본이 반영된다.

### 2. 버전 태깅(Tagging) 및 배포 규칙
* 실시간 배포가 일어나는 메인 소스 코드 변경 시에만 명확하게 버전을 추적하기 위해 **CI/CD 파이프라인(예: GitHub Actions)을 통한 자동 태깅 시스템**을 구축하여 운영한다(현재는 미구축 — 수동 수행).
* **[중요/정책 변경] 태깅 시점은 `main` 머지 후가 아니라 PR을 올리기 직전이다.** 저장소 소유자가 PR을
  승인·머지하는 순간에는 이미 작업 커밋과 버전 태그·`versionlog.md` 기록이 그 브랜치 안에 함께
  들어 있어야 한다 — 머지 직후 별도로 태깅·문서화를 뒤쫓아가지 않는다.
* **태깅 트리거 조건(브랜치 종류별, PR 오픈 직전에 수행):**
  * `epic/*` (또는 `major/*`) ➡️ PR 오픈 직전: **Major(X) 버전 버프** 및 `vX.0.0` 태그를 브랜치 최신 커밋에 생성
  * `feature/*` ➡️ PR 오픈 직전: **Minor(Y) 버전 버프** 및 `vX.Y.0` 태그를 브랜치 최신 커밋에 생성
  * `hotfix/*` ➡️ PR 오픈 직전: **Patch(Z) 버전 버프** 및 `vX.Y.Z` 태그를 브랜치 최신 커밋에 생성
  * `data` ➡️ 애초에 `main` 머지가 없으므로 **해당 없음**
  * `document-archive` ➡️ 애초에 `main` 머지가 없으므로 **해당 없음**
* **[필수]** 버전이 올라갈 때마다(자동 태깅이든 수동 공표든) 그 사실을 리포지토리 루트의
  `versionlog.md`에도 함께 기록한다 — 버전, 날짜, 무엇이 바뀌었는지, 관련 PR 번호를 남긴다.
  자동 태깅 CI가 아직 구축되지 않은 현재는 이 `versionlog.md` 기록이 유일한 버전 이력이므로
  반드시 빠짐없이 남긴다. `versionlog.md`는 A 티어 문서로 `main` 전용이며 `document-archive`
  브랜치에는 존재하지 않으므로, 태깅을 발생시키는 해당 코드 브랜치(`epic/*`/`feature/*`/`hotfix/*`)
  안에서 PR 오픈 직전까지 직접 갱신한다.

## General Rules

* 하나의 브랜치는 하나의 작업(Task)/주제만 담당한다. (단일 브랜치인 `data`와 `document-archive`는 예외적으로 상시 재사용한다.)
* Merge 정책은 `CLAUDE.md`의 Working agreement를 따른다.
* 다음(새로운) 작업은 항상 최신 `main`에서 새로운 브랜치를 생성하여 시작한다.
