# Git Branch Workflow

> Merge 정책(누가 언제 `main`에 Merge하는지)은 `CLAUDE.md`의 "Working agreement"가 원칙이다.
> 이 문서는 그 원칙 위에서 **브랜치 종류별 생성·재사용 규칙**만 다룬다.

## Branch Types

### `feature/*` (또는 `feat/*`)

* 새로운 기능 개발용 브랜치
* 항상 최신 `main`에서 생성한다.
* 기능 하나당 브랜치 하나를 사용한다.
* `main`에 변경 사항이 추가되었다면 Merge 전에 최신 `main`으로 Rebase한다.

---

### `hotfix/*`

* 버그 수정용 브랜치
* 항상 최신 `main`에서 생성한다.
* 버그 하나당 브랜치 하나를 사용한다.
* 수정 완료 후 가능한 한 빨리 PR을 연다(긴급도가 높으므로 리뷰 우선순위도 가장 높다).

---

### `docs/*`

* 문서 작업 전용 브랜치. **주제(topic) 하나당 브랜치 하나**를 사용한다 (예: `docs/handoff-next-steps`,
  `docs/boundary-tension-open-question`) — 문서 전체를 아우르는 단일 `docs` 브랜치가 아니다.
* 같은 주제로 계속 작업 중이면 새 브랜치/새 PR을 만들지 않고, 열려 있는 그 브랜치에 계속
  Commit·Push한다 — Push하면 열려 있는 PR이 자동 갱신되므로 새 PR을 또 열 필요가 없다.
* 주제가 바뀌면 (기존 브랜치가 아직 열려 있어도) 새 `docs/*` 브랜치를 새로 판다.
* `docs/ref/` 이하 문서(특히 `docs/ref/user-opinion/`)는 읽기 전용 참조 자료다 — 사용자의
  명시적 허락 없이 편집·삭제·이동·이름변경하지 않는다 (`CLAUDE.md` 문서 취급 규칙 참조).
* 일반적으로 최신 `main`으로 Rebase할 필요는 없다.
* 단, 문서 내용이 최근 코드/데이터 변경사항을 설명하거나 의존하는 경우에는, Commit 전에
  최신 `main`을 한 번 pull해서 merge하고 작업한다.

---

### `research/*`

* 새로운 알고리즘, 모델, 분석 방법 등을 검증하기 위한 연구용 브랜치
* 항상 최신 `main`에서 생성한다.
* 하나의 연구 주제당 브랜치 하나를 사용한다.
* 실험 코드는 앱에 직접 반영되지 않아도 된다.
* 연구 결과가 유의미하지 않다면 PR을 열지 않고 브랜치를 삭제해도 된다(저장소 소유자와 확인 후).
* 연구 결과가 채택되면 새로운 `feature/*` 브랜치를 생성하여 정식 기능으로 구현한다.
* `research/*` 브랜치를 그대로 `main` 대상 PR로 열지 않는다.

---

## 코드/데이터 vs 문서 — 브랜치 분리 원칙

* **코드·데이터 변경**(로직 수정, CSV/데이터셋 수정 등)은 **항상 별도 브랜치를 새로 파고
  PR도 새로 연다** — 리뷰어가 실제 동작을 검증해야 하므로 문서 브랜치와 섞이면 안 된다.
* **문서 전용 변경**은 위 `docs/*` 규칙대로 같은 주제의 기존 브랜치를 재사용한다.
* 두 종류는 서로 다른 파일을 건드리는 한 독립적으로 병행 작업 가능하다(머지 충돌 없음).

---

## Merge Priority

브랜치의 우선순위는 다음과 같다 (PR 리뷰 요청 시 저장소 소유자에게 안내하는 순서).

1. `hotfix/*`
2. `feature/*`
3. `docs/*`

긴급 수정 PR은 가능한 한 먼저 검토·Merge되도록 우선 안내한다.

---

## Rebase Policy

`hotfix/*`가 `main`에 Merge된 이후에도 작업 중인 `feature/*` 브랜치가 있다면, 가능한 한 빠르게
최신 `main`으로 Rebase하여 충돌을 미리 해결한다.

---

## General Rules

* 하나의 브랜치는 하나의 작업(Task)/주제만 담당한다 (`docs/*`는 "주제" 단위로 예외적 재사용).
* Merge 정책은 `CLAUDE.md`의 Working agreement를 따른다.
* 다음(새로운) 작업은 항상 최신 `main`에서 새로운 브랜치를 생성하여 시작한다.
* `docs/*`만 예외적으로, 같은 주제인 동안 지속적으로 재사용한다.
