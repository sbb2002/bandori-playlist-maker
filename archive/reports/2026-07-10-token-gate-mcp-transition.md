# 티켓: 토큰 게이트 → MCP 관찰 도구 전환 (사용자 결정)

- 일자: 2026-07-10 15:00 KST경
- 결정자: 사용자
- 상태: **미착수** — 기기 전환 후(토큰 초기화 시) 다른 로컬에서 수행

## 목표

`scripts/token_gate.py`의 **트랜스크립트 탐색(token seeking) 방식 집계를 폐기**하고,
MCP 서버 기반의 **관찰(observation) 도구**로 전환한다. 게이트가 수치를 "추정"하지 않고,
신뢰 가능한 소스의 수치를 "그대로 보여주는" 구조로 바꾼다.

## 폐기 사유 (결함 증거)

같은 시각(2026-07-10 14:47~15:00 KST) 실측 비교:

| 항목 | token_gate.py | 실제 `/usage` |
|---|---|---|
| 세션 | **2.0%** | **85%** |
| 주간 | 9.1% | 13% |
| Fable 주간 | 14.5% | 14% |

세션 수치가 완전히 어긋난 원인은 구조적이다:

1. **세션 정의 오류**: 스크립트는 "mtime 최신 트랜스크립트 파일 1개"를 세션으로 간주하지만,
   실제 한도는 **서버측 5시간 롤링 윈도에서 모든 대화를 합산**한다. 새 대화를 열면 스크립트
   수치는 0 근처로 리셋되지만 실제 세션 사용량은 그대로 누적되어 있다 (이번 2.0% vs 85%가
   정확히 이 케이스).
2. 캐시 읽기 토큰 가중치 등 서버측 산정 방식이 미공개라 분모 보정(당일 캘리브레이션)도
   몇 시간 만에 무효화됨.
3. 주간 수치는 근사가 맞는 편이나, HOLD 판정을 좌우하는 것은 세션 축이므로 도구 신뢰 불가.

## 전환 설계 방향 (기기 B에서 구체화)

- **골격**: 워크스페이스의 `claudespace/custom-mcp-server` 템플릿 재사용 (Python 3.13+,
  `mcp>=1.25.0`). 부장(메인 세션)이 MCP 툴 호출로 수치를 조회.
- **수치 소스 후보** (기기 B에서 조사 후 택1, 우선순위 순):
  1. Claude Code `/usage`가 참조하는 서버측 사용량 조회 경로가 프로그래밍적으로 접근
     가능한지 조사 (비공식일 가능성 높음 — 변동 리스크 명시할 것).
  2. Claude Code 공식 OTel 텔레메트리(`CLAUDE_CODE_ENABLE_TELEMETRY=1`,
     `claude_code.token.usage` 메트릭)를 로컬 수집해 노출 — 공식 지원이지만 이것도
     로컬 집계라 세션 윈도 문제를 완전히 해결하는지 검증 필요.
  3. **폴백(확실한 최소안)**: 사용자가 `/usage` 실측치를 MCP 툴로 기록(`record_usage`)하면
     서버가 타임스탬프와 함께 저장하고, 부장이 조회(`get_usage`)할 때 기록 경과시간을 함께
     반환 — "오래된 관찰값" 경고 포함. 추정을 전혀 하지 않으므로 틀릴 수 없는 구조.
- **유지할 것**: HOLD 판정식(`세션 ≥80 or 주간 ≥90 or Fable ≥70`)과 JSON 출력 계약은
  그대로 두고 수치 소스만 교체. `compute_verdict` 등 순수 함수와 테스트는 재사용 가능.
- **폐기할 것**: 트랜스크립트 스캔 로직(`iter_usage_records`, `sum_dedup_tokens`의 세션 축
  사용), `token_gate_config.json`의 분모 캘리브레이션 체계.
- 전환 완료 후 `docs/orgarnization.md` §4·R5 본문 개정 (현재는 §4에 폐기 예고 주석만 추가).
