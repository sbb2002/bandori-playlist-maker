## 결정 — A(OAuth 인증)·B(할당량 증설) 모두 보류, 추가 조치 없음

**A — OAuth 앱 인증(일반 사용자 확대)**
- 막힌 지점: 홈페이지·프라이버시 정책을 본인 소유 도메인으로 옮기고 도메인 소유권을 검증해야
  함. GitHub Pages(Public Suffix List 도메인)로는 불가.
- 확인된 사실: 커스텀 도메인 구매 계획 없음 — 비상업 팬 프로젝트에 추가 비용을 들일 의사 없음.
- 처리: 해당 메일(`19f7491015723fa0`)에 회신하지 않고 무응답으로 보류. 회신해야만 절차가
  진행되는 구조라, 무응답 자체가 안전한 보류 수단.

**B — YouTube Data API 할당량 증설**
- 직접 계산: 기본 10,000유닛/일 기준, 60분 이내 플리(`playlists.insert` 50 + `playlistItems.insert`
  50×곡수 ≈ 900유닛)는 하루 약 11회, 180분(약 50곡, 2,550유닛) 플리는 하루 약 3회까지 가능.
  실사용 패턴(60분 이내 위주)이면 하루 11~12회로 충분하다고 판단.
- OAuth 미인증 상태에서는 애초에 Cloud Console에 등록된 테스트 계정 100명까지만 로그인
  가능해 수동 등록 자체가 더 큰 병목 — quota를 늘려도 실익이 작음("꿔다놓은 보릿자루").
- 처리: 별도 철회 회신 없이 그대로 방치. 이미 보낸 컴플라이언스 답변으로 리뷰 자체는 유지된 채
  둠.

## 서비스 영향 — 없음

- OAuth 미인증 상태에서도 앱은 정상 작동. 테스트 계정 100명(Cloud Console 수동 등록)은
  "내 재생목록에 넣기"로 실제 YouTube 재생목록 저장 가능.
- 그 외 모든 사용자는 v1.6.1의 **익명 임시 재생목록 폴백**(로그인 불필요, YouTube 임시 재생
  링크)으로 핵심 기능(무드 기반 플레이리스트 생성·재생)을 제약 없이 사용 가능.
- 즉 "커뮤니티에 자유롭게 던져놓고 아무나 쓴다"는 원래 목표는 폴백 경로로 이미 달성돼 있고,
  "내 계정에 영구 저장"만 테스트 계정 100명 한정 보너스 기능으로 남음.

- 커뮤니티 사용자가 자유롭게 로그인해서 자기 계정에 저장하는 건 구조적으로 불가능(Google
  정책상 도메인 소유 인증이 전제 조건). 우회로 없음 — 확인 완료. 사용자·개발자 양쪽 모두에게
  불편·꺼림칙한 지점이라는 점 인지하고 감수하기로 함.
