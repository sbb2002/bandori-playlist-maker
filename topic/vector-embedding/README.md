# 가사·음원 피처 벡터 임베딩 검색 (topic/vector-embedding)

## 배경·목적

`topic/mfcc_analysis`의 정성 탐색 결론에서 출발한다: 음향학적 통계 파라미터(f0/jitter/HNR/
화성/에너지 등)가 설명하는 것은 보컬 표현의 극적임 정도까지이고, **장르감·감정 인상의 잔차는
음향 feature만으로는 못 짚는다**(예: 迷星叫 vs Georgette Me, Georgette You — 지표가 비슷한데
들리는 감정이 다름). 그 잔차를 설명할 핵심 축으로 **가사(의미 층위)**가 지목됐고, 이 주제는
그 후속으로 **가사 임베딩 + 음원 피처를 결합한 벡터 검색**으로 선곡(서칭) 퀄리티를 올릴 수
있는지 검증한다.

## 가설 (파이프라인 설계)

상세 다이어그램: [`hypothesis.md`](hypothesis.md) (렌더링: `fig/hypothesis.png`)

1. **Phase 1 — LLM 문맥 확장**: 사용자 자연어 입력을 Groq LLM이 (a) 더 풍부한 감성 서술문으로
   확장하고 (b) 에너지 타겟 배열(예: `[0.3, 0.4, 0.35, 0.2]`)을 추출한다.
2. **Phase 2 — 임베딩·벡터 검색**: 확장된 서술문을 사전학습 문장 임베딩 모델(예: KR-SBERT)로
   벡터화하고, 곡별 가사·오디오 벡터를 저장한 Vector DB에서 유사도 검색으로 1차 감정 후보군
   (~50곡)을 추출한다.
3. **Phase 3 — 에너지 매칭**: 에너지 타겟 배열의 슬롯별로 후보군에서 에너지가 가장 가까운 곡을
   배치해 최종 플레이리스트를 만든다.

기존 앱 구조와의 정합: Phase 3는 구조화된 입력(후보군 + 에너지 배열)에 대한 **순수 함수**로
유지되므로 PRD §8의 hexagonal 제약과 충돌하지 않는다. 임베딩 모델도 LLM처럼 포트 뒤의 교체
가능한 어댑터로 두면 된다. 기존 에너지 진행·Camelot 믹싱 로직은 Phase 3에 그대로 얹힌다.

## 검토 포인트 (승격 시 검토 결과, 2026-07-16 — 연구에서 확인할 것)

1. **언어 불일치 (최우선)**: 가사는 일본어, 프롬프트는 한국어. KR-SBERT는 한국어 단일 언어
   모델이라 일어 가사를 직접 임베딩하면 같은 의미 공간에 놓이지 않는다. 후보 대안:
   - (a) 다국어 임베딩 모델 (BGE-M3, multilingual-E5, LaBSE 등) — 교차언어 검색을 직접 지원.
   - (b) **LLM으로 곡별 "무드 요약문"(한국어)을 생성해 그것을 임베딩** — 프롬프트와 같은
     언어·같은 도메인(감성 서술)으로 정규화되고, 가사 원문 저장 불필요로 저작권 문제도 함께
     회피. mfcc_analysis에서 이미 세운 "원문 DB화 금지, 임베딩만 보관" 원칙과 정합.
   - (c) 가사 번역 후 임베딩 — 비용 대비 이점 불분명.
2. **프롬프트↔가사 도메인 갭**: 사용자 프롬프트는 "듣고 싶은 기분" 서술이고 가사는 서사·1인칭
   내용이다. 밝은 가사의 어두운 곡 같은 어긋남이 존재 — 확장문↔가사 유사도가 실제 무드 매칭으로
   이어지는지가 **이 연구의 핵심 측정 대상**이다. (위 (b)안은 이 갭도 줄여준다.)
3. **가사 확보 경로**: 데이터셋(660곡)에 가사가 없다. Whisper ASR(mfcc_analysis에서 검토 —
   가창 WER·환각 리스크 있음) vs 외부 소스. 어느 쪽이든 원문은 임베딩 변환 후 즉시 폐기.
4. **"가사 및 오디오 벡터" 결합 방식 미정**: 텍스트 임베딩 공간과 오디오 피처는 서로 다른
   공간이라 같은 DB에 넣는다고 합쳐지지 않는다. 가장 단순·안전한 분리안 — **1차 검색(Phase 2)은
   텍스트만, 오디오 피처(energy 등)는 Phase 3 매칭에 사용**. 오디오-텍스트 조인트 임베딩
   (CLAP류)이나 late-fusion 가중합은 별도 method로 분리해 실험.
5. **운영 제약 (채택 시)**: Render free tier(512MB RAM)에 SBERT 상주 곤란 — 프롬프트 임베딩을
   어디서 할지(경량 모델/외부 임베딩 API) 채택 단계에서 결정 필요. 곡 벡터는 오프라인
   사전계산이므로 문제 없음. 또한 660곡 규모에서는 별도 Vector DB가 과설계 — 인메모리
   numpy/FAISS로 충분.
6. **평가 방법**: "서칭 퀄리티 향상"의 측정 기준 필요 — 소규모 수작업 평가셋(프롬프트→기대곡
   페어) 또는 현행 파이프라인과의 정성 A/B. 기존에 계획된 "감정 라벨링 라운드"와 맞물림.
