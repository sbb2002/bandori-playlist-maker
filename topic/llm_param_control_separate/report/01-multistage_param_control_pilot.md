# llm-param-control-separate: 파라미터별 순차 LLM 호출 파일럿 — 요약 보고서

**실행일**: 2026-07-19. **분량**: 정식 실험이 아니라 실사용 파일럿이므로 간략 기록만 남긴다.

## 1. 배경

(`../backgroud.md` 원문 요약) 에너지 제어를 단일 LLM 호출로 처리하는 기존 방식이 배포판 대비
특출난 개선을 못 보였다(`energy_selection` 연구 참고). 한 번의 요청으로 재생시간·구간분할·
구간별 무드·에너지·요약까지 모두 뽑기엔 저비용 모델(`llama-3.1-8b-instant`) 하나에 부담이
크다고 보고, 파라미터마다 순서대로 LLM을 호출해 JSON을 조립하는 방식을 시도했다.

## 2. 구현

`feature/llm-param-control-separate` 브랜치, `GroqMultistageMoodInterpreter`
(`src/backend/app/adapters/groq_multistage_adapter.py`) — opt-in 어댑터
(`MOOD_INTERPRETER=groq_multistage`), 기존 단일호출 `groq_adapter`는 그대로 유지.

4단계 순차 호출:
1. 전체 재생시간(분, 정수)
2. 구간 수(2~5) + 구간별 (길이, 감정 키워드)
3. 구간별 에너지(0.00~1.00)
4. 요약 카드 문장(원래 의도가 이것도 LLM이 쓰게 하는 것이었음 — 결정론적 조립 대신)

실사용 중 발견·수정한 것:
- 무료 모델이 "숫자만 답해" 지시를 자주 어겨 장문 설명을 앞세움 → `===ANSWER===` 마커 기반
  관용 파싱 도입(마커 못 찾으면 "마지막" 숫자/줄로 폴백).
- summary 카드 폴백 문장의 '로/으로' 조사 버그(받침 유무 미반영) 발견·수정.
- 3차 에너지 프롬프트에 현재 필터(밴드 등) 곡 풀의 energy min/max/mean/std 힌트를 추가 —
  실존하지 않는 비현실적 극단값을 LLM이 덜 찍도록 유도.

## 3. 결과

정량 평가(스코어링) 없이 실사용 확인만 진행. 연구자(사용자) 실사용 피드백:

> "확실히 단일 요청으로 처리할 때보다 청취감이 좋아졌어."

에너지 분포 힌트 추가 이후 이 개선이 확인됨. `20b`(`openai/gpt-oss-20b`)로 모델 교체 시도도
있었으나 최종 배포는 `llama-3.1-8b-instant` 유지로 판단(세부 비교는 세션 로그 참고, 별도
정량 기록 없음).

## 4. 결론

PR #51로 `main` 머지 완료(v1.10.0). 후속으로 에너지 분포 힌트를 3차 프롬프트에 추가하는
개선(PR #53, v1.11.0)도 같은 파일럿의 연장선에서 반영됨.

## 5. 다음 단계 (아이디어 수준, 미착수 — 구체 설계는 별도 세션)

3차 제어(현재는 에너지 하나)를 다변수로 확장하는 아이디어:

- 대상 변수(안): **tempo**(BPM), **mood**(가사+음향을 함께 고려한 긍정/부정 극성), **drama**
  (보컬의 극적인 정도), **가사**(내용 반영), 그 외 자체제작 변수 후보.
- 방법(안): 오늘 에너지에 했던 것과 동일하게, 각 변수의 곡 풀 분포(min/max/mean/std)를 LLM
  프롬프트에 제공해 실존 범위 안에서 값을 고르게 유도.
- **주의**: 이 절은 방향성 메모일 뿐이다. 각 변수의 실제 데이터 가용성(예: mood·drama는
  자체 산출 필요, 가사는 별도 파이프라인 필요 — `vector-embedding-phase2-status` 메모리의
  "가사와 음향은 거의 직교" 결과와 상충 여부도 확인 필요)과 구체적 산식·검증 방법은 다음
  세션에서 별도로 설계한다.
