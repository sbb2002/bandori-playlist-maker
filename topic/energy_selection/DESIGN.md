# energy_selection 실험 설계서 — LLM 에너지 파라미터 해석 프롬프트 튜닝

> **위치와 성격**: `chord_progression`·`mood_warmth`·`vector-embedding`·`selection_pipeline`은 모두
> 오디오/가사 신호 기반 음향 지표 연구였다. 이 주제는 다르다 — 오디오 처리도 임베딩도 없고,
> `src/backend/app/adapters/prompt.py`의 `SYSTEM_PROMPT` 한 곳이 실험 대상 전부다. 순수 LLM
> JSON 산출 정확도 문제이므로 **저비용·고속 A/B 루프**(오디오 다운로드·Demucs·bge-m3 불필요)로
> 진행 가능 — 이것이 이 주제를 승격할 만한 핵심 근거다.
>
> **구현자는 이 문서만 보고 코드를 작성한다.**

## 0. 배경 요약 (`background.md` 원문 참조)

배포판에서 사용자가 에너지 진행을 모델(LLM)이 알아서 정하도록 맡기면(수동 그래프 편집이 아닌
자연어 경로), 아래 두 실사례에서 명백히 잘못된 값이 산출됐다.

| 사용자 요청 | 산출 | 문제 |
|---|---|---|
| "장마철 꿉꿉한 기분을 날려줄 노래 **아니면** 습기에 젖는 서정적인 노래로 부탁해." | 3구간 `[0.30, 0.40, 0.50]` | 요청 어디에도 "점점 고조"류 진행 표현이 없는데(오히려 "날려줄"↔"서정적인"이라는 **대조적인 양자택일**) 완만한 상승 아크가 나옴 — 어느 쪽도 명확히 반영 못 한 애매한 타협값 |
| "천천히 워밍업해서 30분즈음에 최고조에 달하게 해주고 그 뒤에 40분까지 마무리하도록 쿨다운하게 도와줘" | 1차: 파싱 실패 / 2차: 3구간 `[0.20, 0.17, 0.15]` | "최고조"(peak) 요청인데 오히려 **단조 감소**, 그것도 전 구간이 낮은 값(0.15~0.20)에 몰림 — 방향·크기 둘 다 틀림 |

## 1. 원인 소재 확정 (코드 감사 결과)

`main` 브랜치 기준(`research` 브랜치는 소스 정리됨, `git show main:<path>`로 확인) 아래 경로를
직접 읽고 확인했다. **결정적 로직에는 버그가 없다** — 문제는 전적으로 `SYSTEM_PROMPT`가 LLM을
충분히 제약하지 못하는 데 있다.

- `src/backend/app/adapters/prompt.py`
  - `SYSTEM_PROMPT`: `stage_energies`(선택 배열, 길이 2~5)에 대해 "운동·유산소·러닝처럼 에너지가
    오르내리는 활동은 **반드시** 자연스러운 아크를 담아라. 예: 러닝=`[0.3,0.7,0.85,0.5]`"라는
    지시만 있다. **①양자택일/대조 표현**(사례1)에 대한 지시가 전혀 없고, **②절대 시간 마커로
    피크·전환 지점을 지정하는 요청**(사례2, "30분에 최고조")을 스테이지 배열의 상대적 위치로
    변환하는 방법도 전혀 가르치지 않는다 — worked example이 상대적 순서(준비→가속→유지→마무리)만
    보여줄 뿐 "N분째"라는 절대시간 개념 자체가 프롬프트에 없다.
  - `parse_mood()`: 클램프·기본값 주입·JSON 관용 파싱뿐, 값을 스무딩/평균하는 로직 없음 —
    사례1의 "애매한 타협값"은 파싱 단계가 아니라 **LLM 산출 자체**에서 나온 것으로 확정.
- `src/backend/app/domain/energy.py`
  - `stage_energy_targets()`: 순수 선형 보간(`start + (end-start)*i/(N-1)`) — LLM이
    `stage_energies`를 안 주고 `start_energy`/`end_energy`/`stage_count`만 줄 때만 쓰이는
    폴백. 로직 자체는 정확하고 버그 없음.
- `src/backend/app/domain/selection.py` `_stage_targets_and_counts()` (L144~159)
  - 우선순위: 수동 UI `stage_specs` > LLM `stage_energies` > LLM `start/end` 선형 폴백.
    LLM이 뭘 주든 그대로 전달만 함 — 여기도 원인 아님.
  - **부수 발견(범위 밖)**: `distribute_counts()`가 스테이지별 곡 수를 **개수 기준 균등 분배**만
    한다. 즉 `stage_energies` 값 자체가 완벽해도 "30분 중 처음 10분 워밍업, 다음 20분 빌드업,
    마지막 10분 쿨다운"처럼 **스테이지 길이가 불균등**해야 하는 요청은 현재 아키텍처로는 못
    담는다(각 스테이지가 곡 수 기준 거의 같은 시간을 차지). 이건 프롬프트 튜닝으로 못 고치는
    구조적 갭이라 **이번 실험 범위 밖**으로 명시하고 Phase 2 후보로만 남긴다.
- `src/backend/app/repo/song_repo.py` `intensity()`: 곡 강도 자체는 이미 `energy_full`+
  `i_min/i_mean/i_end` power-mean 합성(검증됨, [[acoustic-feature-audit]]) — **타겟 생성(LLM)과
  타겟-곡 매칭(song_repo)은 서로 다른 계층이며, 이번 실험은 전자만 다룬다.**
- 배포 모델 확정: `main.py` `_build_interpreter()` → `GROQ_API_KEY` 있으면 Groq,
  기본 모델 `llama-3.1-8b-instant`(`groq_adapter.py` `DEFAULT_MODEL`, env `GROQ_MODEL`로 override
  가능). 소형·경량 모델이라 복합 지시 이행력이 약할 수 있다는 점도 실패 재현 가능성을 뒷받침한다.

**결론**: 순수 프롬프트 엔지니어링 문제로 국한된다. 오디오/임베딩 파이프라인 불필요 → 저비용·
고속 반복이 가능해 실험 타당성이 높다. **정식 연구주제로 승격.**

## 2. 연구 질문과 판정 기준 (사전 등록 — 결과 확인 후 변경 금지)

**RQ1(주)**: 실패모드 ①(양자택일/대조 요청)·②(절대시간 피크 지정 요청)에 대한 가이드를
`SYSTEM_PROMPT`에 추가한 개선안이, 현재 배포 프롬프트보다 규칙기반 채점에서 유의하게 나은가?

**RQ2(부, 회귀 방지)**: 개선안이 기존에 이미 잘 동작하던 케이스(단순 상승/하강/일정, 능동적
`stage_count` 조정)에서 성능을 유지하는가?

**판정표**:

| 판정 | 조건 |
|---|---|
| **채택(배포 반영 권고)** | 카테고리 A+B(실패모드) 평균 점수: 개선안 > 배포(Wilcoxon p<0.05 또는 최소 방향 일관 + 효과크기 확인) **AND** 카테고리 C+D(회귀) 평균 점수: 개선안이 배포 대비 0.05점 이상 떨어지지 않음 |
| **부분 채택** | A+B는 개선되나 C/D 중 한 카테고리에서 회귀 — 회귀 원인이 되는 지시문만 재수정해 재검증 |
| **기각** | A+B 개선 없음(배포와 유의차 없거나 악화) — 프롬프트 튜닝으로 해결 안 되는 문제로 재분류(예: 모델 자체 교체 검토로 이관) |

**사전 선언한 한계**: 규칙기반 채점은 "의미적으로 그럴듯한가"까지는 못 본다(예: 값 크기가
맞아도 실제 청취 시 이질감이 있을 수 있음) — 채택되더라도 배포 전 소규모 실사용 스팟체크 권장.

## 4. 채점 방법 (규칙기반, 사람 청취·LLM 판정 불필요)

각 (쿼리 × 프롬프트 variant) 조합을 **n=5회 독립 호출**(배포와 동일 온도 설정)해 파싱 성공률과
값의 변동성(flakiness — 사례2의 1차 파싱 실패 같은 불안정성)을 함께 측정한다.

각 트라이얼에서 `stage_energies`가 있으면 그대로, 없으면 `stage_energy_targets(start, end,
stage_count)`로 재구성해 길이 N의 배열로 정규화한 뒤 아래 지표를 계산한다(쿼리 카테고리별로
적용 지표가 다름):

1. **parse_success** (전 카테고리): JSON 파싱·필수 필드 존재 성공 시 1, 실패 시 0(실패 시 이후
   지표는 전부 0 처리).
2. **shape_match** (전 카테고리): 배열의 연속 차분(diff) 부호열을 기대 shape과 비교.
   - `flat`: 모든 diff의 절대값이 허용폭 이내(§3의 max-min ≤0.15) → 1, 아니면 0.
   - `rising`/`falling`: diff 부호가 전부 기대 방향과 일치(작은 잡음 tolerance ±0.03) → 1,
     대체로 일치(80%+ 구간) → 0.5, 반대 방향 → 0.
   - `peak`/`rise-then-fall`: 최댓값 인덱스가 처음도 끝도 아니고, 그 앞뒤로 각 1회 이상 상승·
     하강 존재 → 1, 최댓값이 끝에서 두 번째 이내로 근사 → 0.5, 단조증가/감소(=peak 실패, 사례2
     유형) → 0.
3. **peak_position_match** (B 카테고리만): 쿼리에 명시된 `(peak_minute / target_minutes)`와,
   산출 배열의 최댓값 인덱스 위치 `idx/(N-1)`의 절대오차. ≤0.2 → 1, ≤0.35 → 0.5, 그 외 0.
4. **magnitude_sanity** (전 카테고리, 쿼리별 사전 정의 임계): 예) B1 최고조 값 ≥0.65,
   워밍업 시작값 ≤0.35 / C1 전 구간 ≤0.35 / C2 전 구간 ≥0.6. 충족 시 1, 아니면 0.
5. **query_score** = 적용 지표들의 평균(트라이얼당) → 5트라이얼 평균·표준편차를 쿼리 최종 점수로.

**variant 간 비교**: 쿼리 단위 대응비교(baseline vs candidate) Wilcoxon signed-rank
(`selection_pipeline` 관례 승계) + 평균차·표준편차 리포트. 표본이 작으므로(쿼리 10개) 유의성보다
방향 일관성·효과크기를 1차 판단 근거로 삼고, p-value는 보조 지표로만 쓴다(§2 판정표에 이미 반영).

## 5. 프롬프트 variant 정의

- **baseline**: `src/backend/app/adapters/prompt.py`의 현재 `SYSTEM_PROMPT` 그대로(vendoring —
  `prod_snapshot` 관례처럼 파일을 그대로 복사해 실험 디렉토리에 고정, 배포 코드는 건드리지 않음).
- **candidate_A**: baseline + 양자택일/대조 요청 지시 추가.
  ("사용자가 '~아니면 ~', '~든 ~든 상관없이'처럼 서로 다른 방향을 나열하며 하나를 고르라는 명시가
  없으면, 무리해서 그 사이의 새로운 진행(상승/하강)을 만들어내지 말고 공통 강도 수준으로 수렴시켜
  flat에 가깝게 두어라.")
- **candidate_B**: baseline + 절대시간 피크 매핑 worked example 추가.
  ("사용자가 특정 분(分)에 최고조/전환을 지정하면, `target_minutes` 대비 그 시점의 비율을
  계산해 `stage_energies` 배열에서 해당 비율 위치의 값을 가장 높게/낮게 배치하라. 예:
  40분 중 30분에 최고조 → 비율 0.75 → 4구간이면 3번째 값 근처를 최고점으로.")
- **candidate_AB**: A+B 동시 적용 — 본 실험의 채택 후보.

## 7. 실행 순서

```
01_run_variants.py → 4 variant × 10쿼리 × 5트라이얼 = 200회 LLM 호출 → out/raw_responses.csv
02_score.py        → out/scored.csv (규칙기반 채점, 사람 개입 없음)
03_compare.py      → variant별 카테고리 평균 + Wilcoxon → §2 판정표 적용 → 최종 결론 문서화
```

호출 200회는 Groq 무료 티어 기준으로도 수 분~수십 분 내 완료 가능한 규모(다른 topic들의
오디오/임베딩 파이프라인과 달리 GPU·다운로드 불필요).

## 8. 알려진 한계 (착수 시점에 이미 아는 것)

- 쿼리 10개(카테고리당 2~4개) — 방향성 탐색 수준, 확증 아님. 채택되더라도 배포 전 실사용
  스팟체크 권장(§2에 이미 명시).
- 규칙기반 채점은 "수치 정합성"만 보고 "청취 시 자연스러움"은 못 잡는다.
- 스테이지 **길이**(곡 수 배분)의 시간축 불균등 문제(§1 부수 발견)는 이번 실험 범위 밖 — 채택
  후에도 "30분에 피크"류 요청이 완벽히 재현되진 않을 수 있음(값의 방향·위치는 맞아도 구간 길이가
  균등 분배라 청감상 밀도가 다를 수 있음). 필요 시 Phase 2로 `distribute_counts()`에 시간비율
  가중치를 추가하는 별도 후속 연구 제안.
- 소형 모델(`llama-3.1-8b-instant`) 자체의 지시 이행 한계일 가능성 — candidate_AB로도 개선이
  안 되면(§2 기각), 프롬프트가 아니라 모델 교체(`GROQ_MODEL` env)를 다음 가설로 검토할 것.

## 9. 후속 실험: 극단값 활용(Phase 2, 2026-07-18 추가)

> `note/additional_experiment.md`에서 제안된 사람 청취평가 그리드 실험은 채택하지 않는다 —
> 연구제안자가 작성 과정에서 "이미 잘 동작한다"고 판단해 기각(에너지값-곡 매칭 자체는 문제가
> 아님). 대신 아래 신규 실패모드가 §0~8과 같은 순수 프롬프트 A/B 방법론으로 재정의됐다.

**신규 실패모드("중심값 수렴", central-tendency bias)**: 사용자의 감정·상황 표현이 극단적이어도
(매우 지침/위로 필요, 인터벌 스퍼트 등) LLM이 `start_energy`/`end_energy`/`stage_energies` 값을
0.4~0.6 근처 중간대로만 산출하는 경향. §0~8의 실패모드(①양자택일, ②절대시간 피크)와 달리 *모양
(shape)*이 아니라 *절대 크기의 극단 활용* 문제라 별도 candidate로 분리한다.

**RQ3**: 극단값 활용 지시(`candidate_E`)를 추가하면, 위로형(카테고리 E)·인터벌형(카테고리 F)
쿼리에서 baseline보다 유의하게 더 극단적인 값을 산출하는가?

**RQ4(회귀 방지)**: `candidate_E`가 기존 카테고리 A~D(§3)에서 성능을 유지하는가?

**신규 쿼리(카테고리 E/F)**:

| id | 카테고리 | 예시 쿼리 | 기대 shape | magnitude_checks |
|---|---|---|---|---|
| E1 | 극단값(위로형) | "오늘 너무 힘들었어... 나한테 위로가 되는 플리 짜줘." | rising | start_below 0.15, end_below 0.65 |
| E2 | 극단값(위로형) | "지친 하루 끝에 마음 편해지는 노래로 채워줘. 처음엔 그냥 무너져도 괜찮으니까." | rising | start_below 0.10, end_below 0.65 |
| F1 | 극단값(인터벌 스파이크) | "5km 동안 조깅할 때 들을 플리 짜줘. 중간에 한번 빠르게 뛸때 쯔음 도파민을 터뜨려줘." | peak | peak_above 0.90, first_below 0.55, end_below 0.55 |
| F2 | 극단값(인터벌 스파이크) | "헬스장에서 웨이트 하는데 세트 사이사이 짧게 전력질주하듯 강하게 터지는 구간 하나만 넣어줘. 나머진 무난하게 가줘." | peak | peak_above 0.90, first_below 0.55, end_below 0.55 |

카테고리 E/F는 `peak_minute`/`target_minutes`를 지정하지 않는다(§0~8의 B/F 구분: B는 "위치
정확도", F는 "극단값 도달 여부"만 검증 — 스코프를 분리해 지시문 간 간섭을 줄인다).

**candidate_E 정의**: baseline + 극단값 활용 지시. "감정·상황 표현이 극단적이면 0.4~0.6 중간대로
뭉뚱그리지 말고 실제 극단값(0.00~0.15 또는 0.85~1.00)을 쓰라"는 지시 + 위로형/인터벌형 worked
example 2개(원문은 `prompts.py`의 `CANDIDATE_E_ADDITION`). `candidate_A`/`candidate_B`와 결합하지
않는다 — 이번 라운드는 candidate_E 단독 효과만 본다(결합은 채택 시 후속 검토).

**채점 확장(§4에 추가)**: `magnitude_checks`에 `end_below`(마지막 값 ≤ threshold) 체크 타입을
추가. 그 외 `shape_match`(rising/peak, 기존 로직 그대로) 재사용, 사람 청취 불필요.

**판정 기준**: §2 판정표와 동일한 논리를 candidate_E에 적용하되, "A+B"→"E+F"(실패모드),
"C+D"→"A+B+C+D"(회귀 방지, 기존 전체 카테고리로 확대 — candidate_E는 새 지시라 기존에 이미
검증된 4개 카테고리 전부에 회귀가 없는지 확인해야 함)로 대체한다.
