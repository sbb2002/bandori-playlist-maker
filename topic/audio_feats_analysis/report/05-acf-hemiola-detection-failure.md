# ACF 기반 헤미올라(3:2) 판별 4연속 실패 기록

`report/02`(method-2, "범위 내 최대 후보" 규칙, RMS 23.61 / Accuracy1 92.5%)의 잔여 오차
18곡을 들여다보니 전부 `drum_tempo_bpm`이 정답의 정확히 1.5배 또는 0.667배(3:2 비율,
헤미올라)로 어긋나 있었다. 2ⁿ 옥타브 후보로는 원천적으로 못 잡는 오차라, 후보군에
{2/3, 3/2}를 추가하고 ACF(onset-envelope 자기상관)로 판별을 시도했다. 결과: **4가지
서로 다른 ACF 활용 방식 전부 실패**, 최종적으로 ACF 기반 판별 자체를 폐기했다.

## 1. 배경 — 왜 헤미올라 후보가 필요했나

18곡의 오차가 전부 1.497~1.514배(≈3:2) 또는 그 역수에 몰려 있었다 — 옥타브(2배/절반)
오차가 아니라 **박자표 오인식**(3박 계열을 4박으로, 또는 반대로 잘못 엮음)으로 진단됐다.
`drum_tempo_bpm`은 드럼 스템에서 `librosa.beat.beat_track`으로 뽑은 값이라, 박자표 가정
없이 순수 onset 간격 규칙성만으로 박을 잡는 이 알고리즘의 구조적 한계로 보인다.

## 2. 상한선 확인 (오라클, 치팅 — 진단 전용)

실전 판별 전에 "후보군에 정답이 있는가"부터 확인했다(`official_bpm`을 미리 알고 최적
후보를 고르는 치팅 계산, 배포 불가 — 상한선 확인용):

| 후보군 | 오라클 RMS(BPM) | 오라클 Accuracy1 |
|---|---|---|
| 2ⁿ만 | 17.78 | 94.76% |
| 2ⁿ + 헤미올라(2/3, 3/2) | **7.70** | **97.03%** |

후보군을 넓히면 이론 상한이 크게 오른다 — 문제는 순전히 "판별 신호"에 있었다.

## 3. 진단 — ACF 크기는 정답과 사실상 무관

5개 후보(2ⁿ×3 + 헤미올라×2)를 ACF 크기로 순위 매겼을 때, **정답 후보가 1등(argmax)일
확률 = 48.7%**(573곡 중 279곡) — 거의 동전 던지기 수준. `drum_tempo_bpm` 자체가 이미
beat_track이 "이 지점의 ACF가 가장 강하다"고 판단해서 나온 값이라, 같은 onset envelope로
ACF를 다시 재는 건 구조적으로 순환 참조에 가깝다는 게 원인으로 추정된다.

## 4. 4연속 실패

| 시도 | 방식 | n | RMS(BPM) | Accuracy1 |
|---|---|---|---|---|
| **baseline(method-2)** | 2ⁿ 범위제한 + 최대후보(ACF 미사용) | 573 | **23.61** | **92.5%** |
| 구 method-3 | 5후보(2ⁿ+헤미올라, [85,220] 범위제한) argmax ACF | 573 | 66.67 | 48.5% |
| method-3(재실험, 치팅 없음) | 2ⁿ(n=-2..2) **범위제한 없이** argmax ACF | 573 | 88.90 | 18.5% |
| method-4 | method-3 + 마진 기반 헤미올라 폴백(임계값 스윕 0.01~0.20) | 573 | 89.04(최선 τ=0.01) | 17.1%(최선) |
| method-5 | baseline + ACF 비교로 헤미올라 무조건 전환 | 573 | 32.15 | 88.66% |

- **구 method-3 → method-3 재실험**: 범위 제한(장르 관측 BPM [85,220])을 없애자 오히려
  **더** 나빠졌다(48.5%→18.5%) — 범위 제한이 ACF의 저주파(느린 후보) 편향을 억누르는
  안전장치 역할을 하고 있었다는 뜻. 선택 배율 분포: `0.25x` 45곡, `0.5x` 145곡, `1.0x`
  374곡, `2.0x` 단 9곡 — 정답의 59%가 ×2 쪽에 있는데도 거의 선택 안 됨.
- **method-4**: method-3의 ACF argmax 선택에 대해 "1위-2위 마진이 작으면(확신 부족)
  헤미올라로 전환"하는 폴백을 얹었지만, 애초에 method-3 자체가 신뢰 불가라 폴백을 얹어도
  구제 불가. 모든 임계값에서 baseline보다 나쁨.
- **method-5**: 가장 정교한 시도 — 이미 검증된 baseline(method-2) 선택값에 대해서만,
  ACF로 헤미올라가 더 커 보이면 전환. 그럼에도 baseline보다 나빠졌다(92.5%→88.66%).
  원인 분석: 알려진 헤미올라 18곡 중 8곡만 잡아냈고(recall 44%), 이미 맞았던 555곡 중
  35곡을 잘못 전환시켰다(정밀도 8/43≈19%) — 손실이 이득보다 컸다. **이건 method-5가
  "실패한 곡만 골라 재시도"한 게 아니라 "모든 곡에 무차별 재판정"을 적용했기 때문**이다
  — "이 곡이 이미 맞았는지"를 아는 것 자체가 정답을 알아야 하는 문제라 회피 불가능했다.

## 5. 결론

1. **ACF(onset-envelope 자기상관)는 이 태스크의 옥타브/헤미올라 판별 신호로 쓸 수 없다** —
   비율 비교(report/02 τ규칙), 전체 argmax, 범위제한 유무, 이진 비교(method-5) 등
   구성을 4번 바꿔가며 실험했지만 전부 baseline보다 나쁘거나 실패했다.
2. **근본 원인은 순환 참조**로 추정된다 — `drum_tempo_bpm` 자체가 beat_track이 이미
   ACF 최댓값 지점을 골라 나온 값이라, 같은 신호로 재판정하면 새로운 정보가 없다.
3. **후보군 확장(2/3, 3/2 추가) 자체는 유효한 아이디어**였다(오라클 Acc1 97%) — 실패는
   순전히 판별 방법(ACF)의 한계였다.
4. **method-2(baseline)가 최종 채택 유지** — RMS 23.61, Accuracy1 92.5%로 5개 시도 중
   부동의 1위. 헤미올라 18곡은 bestdori 매칭곡이면 이미 `bpm_final.csv`가 공식값을
   써서 실질 피해 없음. 미매칭 88곡 중 헤미올라 해당분만 잔여 한계로 남는다.
5. **후속 방향(미착수)**: ACF가 아닌 다른 신호(다운비트 강세 패턴, 코드 변화 주기 등
   실제 3박/4박 리듬 구조를 반영하는 지표)로 재도전하거나, 신호 기반 판별을 포기하고
   bestdori 매칭 범위 확대에 집중하는 두 갈래가 남아 있다.
